# 写作风格指南

本指南约束《AI Platform / Tech Lead Interview Handbook》所有章节的写作。

---

## 一、每题固定模板

每道题严格使用以下结构，用 `---` 分隔每个环节：

```
## Q[N]. 问题标题

**🎤 面试官**

> 提出问题（口语化，像真实面试）。

**🙋 候选人回答**

完整回答。不是背书，是讲经历、讲思考。

---

**🎤 面试官追问**

> 追问（针对候选人回答中的某个点深入）。

**🙋 候选人回答**

继续回答。

---

**🎤 面试官继续追问**

> 再追问（逼近设计本质或边界情况）。

**🙋 候选人回答**

继续回答。

---

### 🏗 架构分析

- **为什么这么设计？**
- **为什么不用其它方案？**（列出 2~3 个替代方案并对比）
- **有哪些权衡？**
- **未来如何演进？**

### 🎯 面试官真正考察什么

一针见血指出考察点。

### ❌ 常见错误回答

候选人最容易犯的错（背概念、答非所问、缺乏思考）。

### ✅ 推荐回答

精炼版完整回答，可作为面试现场参考。

### 📚 延伸知识

相关技术点、推荐阅读、可展开的方向。
```

---

## 二、统一回答原则（六要素）

每个候选人回答都要覆盖（不必显式列编号，但内容要到位）：

| # | 要素 | 说明 |
|---|------|------|
| ① | 为什么会产生这个需求 | 背景、痛点、触发点 |
| ② | 为什么不用其它方案 | 至少对比 1~2 个替代方案 |
| ③ | 为什么选择当前方案 | 决策理由 |
| ④ | 有哪些优缺点 | 诚实分析，不回避缺点 |
| ⑤ | 如何扩展 | 演进路径 |
| ⑥ | 如果重新设计会怎么做 | 反思与迭代 |

---

## 三、内容原则

1. **来源于真实项目**：AI漫剧平台、视频生产平台、Task Platform、AI Platform 等
2. **不写八股**：不问"Redis 是什么"，问"为什么 Task Center 选择 BullMQ"
3. **基础知识自然融入**：Redis/MVCC/Fiber 等概念在项目问题中自然出现
4. **讲思考过程**：不直接给结论，讲怎么推导到这个结论
5. **有方案对比**：架构分析必须列出替代方案

---

## 四、格式规范

- 每章一个 `.md` 文件，命名 `chapter-NN-名称.md`
- 题目编号：`Q1.` `Q2.` ...（章内连续编号）
- 代码块标注语言（```ts / ```sql / ```bash 等）
- 表格用于方案对比
- 每题之间用 `---` 分隔
- 章节标题用 `#`，题目标题用 `##`，分析子标题用 `###`
- 口语部分用 `>` 引用块呈现面试官的话

---

## 五、完成定义（Definition of Done）

一章算"完成"需满足：
- [ ] 所有题目按模板写完（含追问链）
- [ ] 每题六要素到位
- [ ] 架构分析有方案对比
- [ ] 无八股内容
- [ ] 章节内有导语和本章总结
- [ ] README 索引状态更新为 ✅
