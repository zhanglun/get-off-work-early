## 〇、一个心智模型：从"感觉驱动"到"数据驱动"

Day 1-5 一直在凭感觉判断 Agent 好不好——"跑了一下，感觉还行"。但"感觉 OK"在工程上等于"不知道好不好"。Day 6 是从感觉到数据的转折：

```
Day 1-5：做出一个能用的 Agent        （造东西）
Day 6：  量化这个 Agent 到底有多好    （测东西）
Day 7：  部署一个被验证过的 Agent     （上东西）
```

**为什么要评估**，三个真实场景：

| 场景 | 不评估会怎样 |
|------|------------|
| 改了 prompt | 不知道变好还是变差 |
| 换了模型 | 不知道该不该换 |
| 对外宣称"我的 Agent 很强" | 拿不出数据，没说服力 |

> 🔑 一句话：**没有评估的优化，等于闭着眼睛调参。** Day 6 给 Agent 装上"体检系统"。

## 一、评估什么：硬指标 vs 软指标

Agent 不像普通程序（输入→输出对错分明）。它有多个维度，按"是否易量化"分两类：

### 🎯 硬指标（机器可测，客观）

| 指标 | 怎么算 | 数据来源 |
|------|--------|---------|
| **成功率** | `status == "finished"` 的比例 | `ResearchState` |
| **平均步数** | `state.steps` 均值 | `ResearchState` |
| **平均耗时** | 总耗时 / case 数 | 计时 |
| **工具调用次数** | `tool_history` 长度均值 | `ResearchState` |
| **工具成功率** | 每个工具的 success 比例 | `tool_history` |
| **工具选择匹配** | 实际用的 vs 预期用的 | benchmark 定义 |

### 🌊 软指标（需要判定，半客观）

| 指标 | 怎么算 | 难点 |
|------|--------|------|
| **报告相关性** | 是否回答了课题 | 主观 |
| **报告准确性** | 是否基于素材（没编造） | 需核对 |
| **报告完整性** | 关键点是否覆盖 | 主观 |
| **报告简洁性** | 是否啰嗦 | 主观 |

**Day 6 的关键决策**：硬指标全做（容易、客观），软指标用 LLM 互评（见第三章）。

## 二、Benchmark 课题集：评估的基石

### 2.1 为什么不能随便拿问题测

评估 Agent 不能随便拿几个问题跑，要有精心设计的"标准题库"。否则：
- 评估结果不可复现（每次题目不同）
- 无法对比（这次跑 A 题下次跑 B 题，没法说"变好了"）
- 覆盖不全（只跑简单题会高估 Agent）

### 2.2 4 档难度的设计

```python
@dataclass
class TestCase:
    id: str                      # case_001
    topic: str                   # 课题
    difficulty: str              # easy / medium / hard / edge
    expected_tools: List[str]    # 预期会用到的工具
    notes: str                   # 考察点
```

| 难度 | 考察什么 | Day 6 的例子 |
|------|---------|------------|
| **easy** | 基本搜索能力 | "Python 是什么" |
| **medium** | 多角度/该深读时深读 | "React 和 Vue 的区别" |
| **hard** | 复杂对比/技术深度 | "GLM-4 和 GPT-4 的架构区别" |
| **edge** | 异常输入的降级 | "不存在的框架 XYZ" |

**关键设计**：每个 case 标注 `expected_tools`（预期工具）。评估时对比"实际用的 vs 预期的"，算出**工具选择匹配率**——这是判断 Agent 决策质量的核心指标。

### 2.3 edge case 的价值

edge case（乱码、不存在的东西）看起来"没意义"，其实是**最有价值的测试**：

```
case_009（乱码）：
  Agent: 0 步、0 工具，直接说"无法研究"
  judge: 4.5/5 高分

→ 诚实降级比编造答案更值钱。这验证了 Day 4 健壮性设计的价值。
```

## 三、LLM as a Judge：用模型评估模型

### 3.1 软指标的难题

报告质量是主观的，怎么自动评？
- **人工打分**：准确但太慢，10 份报告要半小时
- **规则匹配**：不够灵活，没法判断"写得好不好"
- **LLM 互评**：折中方案——让 LLM 当裁判

### 3.2 LLM as a Judge 模式

> **用一个 LLM 给另一个 LLM 的输出打分。** 这是业界主流的评估方法。

```python
# judge.py 核心
def judge_report(topic, report, findings, model=None):
    prompt = f"""
    你是严格的报告评审员。
    课题：{topic}
    报告：{report}
    素材：{findings}  ← 用于核对是否编造
    
    按 4 维度打分（1-5）：relevance / accuracy / completeness / conciseness
    """
    # 调 LLM，强制 JSON 输出打分
```

**关键点**：给 judge **同时提供报告和素材**。这样 judge 能核对"报告里的内容素材里有没有"——这就是准确性的判定依据。

### 3.3 judge 模型可配置

```python
DEFAULT_JUDGE_MODEL = os.environ.get("EVAL_JUDGE_MODEL", "glm-4-flash-250414")
```

- 默认用 **GLM-4-Flash 自评**（免费，但有"给自己人打高分"偏差）
- 可换 **GLM-4-Plus 互评**（更强模型当裁判，更客观，但付费）

**权衡**：学习阶段先用免费的 Flash 自评，知道有偏差就行；正式评估用更强的模型当裁判。

### 3.4 LLM 互评的局限

| 局限 | 表现 | 缓解 |
|------|------|------|
| **自评偏差** | Flash 给 Flash 打分偏高 | 换更强模型当 judge |
| **打分波动** | 同份报告两次打分不同 | 低温度（0.1）+ 多次取均 |
| **无法替代人工** | 微妙的质量差异 LLM 分不清 | 关键场景仍需人工复核 |

## 五、踩坑记录（真实遇到的）

### 🕳️ 踩坑 1：metrics.py 的 generator bug

**现象**：报告生成时报 `TypeError: unsupported format string passed to generator.__format__`。

**原因**：分难度统计时写错了：
```python
# ❌ 错误：mean(生成器表达式 + 外层运算)
mean(r.scores["relevance"] + ... for r in diff_scored)  # 把整个当成了一个值

# ✅ 正确：先对每个 r 算均分，再对均分取 mean
mean(
    (r.scores["relevance"] + ...) / 4
    for r in diff_scored
)
```

**教训**：Python 的 `statistics.mean()` 接收可迭代对象，写生成器表达式时**括号位置很关键**。算"每项均分的均分"要先算内层再算外层。

### 🕳️ 踩坑 2：DuckDuckGo 网络波动导致评估耗时剧变

**现象**：第一次跑 case_001 单个用了 **265 秒**（4 次搜索只成功 1 次）；第二天跑全套 10 个只要 **372 秒**（平均 37 秒/个）。

**原因**：DuckDuckGo 通过代理的稳定性受网络环境影响极大。Day 3 笔记的踩坑 3 提过偶发不稳定，这次是更严重的"同一天内稳定性差异 7 倍"。

**教训**：**外部免费服务的性能波动会严重干扰评估**。评估结果里的"耗时"指标，必须标注网络条件，否则没有可比性。这也是为什么生产评估要用付费搜索（稳定）。

### 🕳️ 踩坑 3：prompt 优化有连锁反应（重要发现）

**现象**：针对评估发现的"Agent 偷懒不搜索"（case_005/006），我加了"必须至少搜索一次"的硬约束。重跑后发现：
- ✅ **目标 case 修复了**：case_005/006 确实开始搜索了，分数 4.0→4.2
- 🔴 **但其他 case 反而变差了**：case_002/008 优化前会搜索，优化后反而不搜了

**原因**：这是 LLM 的**随机性**（`temperature=0.3`）。我改的 prompt 和 case_002/008 变差没有因果关系——同一版本的 Agent，两次跑同一个 case，结果都可能不同。

**教训**：**prompt 优化不是线性改进，有连锁反应和噪音**。单次评估不可靠，判断"优化是否有效"必须多次评估取平均。详见第六章。

## 六、难点与思考

### 思考 1：LLM 评估的天生噪音（最重要的认知）

这次实验最深刻的发现：**同一个 Agent、同一个课题、同一份代码，两次运行结果可能差异很大。**

证据（优化前后对比，同一版本 Agent 的不同 run）：

| 课题 | 第一次 run | 第二次 run |
|------|-----------|-----------|
| HTTP/HTTPS 区别 | search_web / 4.8 | 不搜索 / 4.0 |
| GLM-4 vs GPT-4 | search+fetch / 4.2 | 不搜索 / 4.0 |

**这些差异不是 prompt 改动造成的，是 LLM 的随机性。**

**后果**：单次评估的结论不可靠。你看到"分数从 4.8 掉到 4.0"，可能只是噪音，不是真的退步。

**正确做法**：
- 每个 case 跑 **3-5 次取平均**
- 对比时要看**统计显著性**，不是单次差异
- 评估系统应支持 `--repeat N` 参数

> 这是 Day 6 给 Day 7+ 留的关键教训：**LLM 评估必须去噪**。业界做法包括多次采样、不同温度、集成评估等。

### 思考 2：prompt 优化是"全局手术"

改 prompt 时容易有错觉："我只改了 X，应该只影响 X。" 实际上 prompt 是 LLM 的"完整世界观"，改一处会**重新分配 LLM 的注意力**。

```
我想强化"必须搜索"：
  + 加了硬约束（正面）
  - 但 LLM 的注意力被"搜索"占走，对其他指令的遵循度下降（负面）
  → 连锁反应：没动过的 case 行为也变了
```

**对比传统编程**：改一个函数只影响这个函数的调用方。改 prompt 影响的是 LLM 的**所有**决策——因为它每次都读完整 prompt。

**实践建议**：
- prompt 改动要**小步快跑**，一次只改一处
- 每次改动后**必须重跑 benchmark**
- 关注"综合指标"而非单个 case

### 思考 3：评估的价值不在"打分"，在"发现问题"

这次评估最有价值的不是"4.3/5"这个分数，而是发现的具体问题：

| 评估发现 | 没评估会怎样 |
|---------|------------|
| case_005/006 偷懒不搜索 | 根本不知道有这问题 |
| 完整性是短板（4.1） | 以为"写得挺全" |
| edge case 诚实降级得高分 | 不知道"诚实"是优势 |
| search_web 成功率 100%（这次） | 不知道网络稳定性影响 |

**评估的真正价值是"照亮盲区"**——你以为 OK 的地方可能有问题，你以为差的地方可能其实不错。

### 思考 4：硬指标和软指标要分开看

| 指标类型 | 特点 | 作用 |
|---------|------|------|
| 硬指标 | 客观、可复现 | 判断"跑得快不快、稳不稳" |
| 软指标 | 主观、有噪音 | 判断"做得好不好" |

**关键**：硬指标出问题 = 架构/工具有 bug；软指标出问题 = prompt/逻辑要调。两类问题诊断方式不同，不能混为一谈。
