# RAG 学习笔记（Lesson 04）

> RAG = Retrieval-Augmented Generation（检索增强生成）
> 让 Agent 能读私有文档——从"只能搜互联网"到"能读知识库"
> 技术栈：sentence-transformers + Chroma + 智谱 GLM

---

## 〇、心智模型：从"搜互联网"到"读私有文档"

```
Day 1-10 的 Agent：只能搜互联网（公开信息）
+ RAG：           能读私有文档（企业刚需）
```

**为什么需要 RAG**：LLM 的知识有截止日期，也不知道你的私有数据。公司报销流程、团队代码规范、产品文档——这些互联网上搜不到，必须靠 RAG。

> 🔑 一句话：**RAG = 先从你的文档里检索相关内容，再让 LLM 基于内容回答。**

---

## 一、RAG 的四个核心步骤

```
① 文档处理    ② 向量化        ③ 检索         ④ 生成
─────────    ─────────     ─────────    ─────────
PDF/TXT      文本 → 向量     问题 → 向量    检索结果 + 问题
→ 切块        （Embedding）   → 找最相似的    → 喂给 LLM
              → 存向量库      几个段落       → 生成答案
```

| 步骤 | 做什么 | 我们的实现 |
|------|--------|-----------|
| ① 文档处理 | 解析 + 切块 | `loader.py`（滑动窗口，chunk_size=400, overlap=50） |
| ② 向量化 | 文本 → 数字向量 | `embedder.py`（bge-small-zh，512 维） |
| ③ 检索 | 找最相似的块 | `store.py`（Chroma 向量库） |
| ④ 生成 | LLM 基于检索回答 | `chain.py`（prompt + GLM） |

## 二、为什么切块（Chunking）

```
一份 500 页文档 = 可能 20 万字 = 30 万 token
→ 直接塞 prompt？撑爆上下文窗口
→ 切成小块 → 只取最相关的几块 → 刚好
```

### 切块策略：滑动窗口 + 重叠

```
原文：ABCDEFGHIJ...（chunk_size=5, overlap=2）
块1：ABCDE
块2：DEFGH     ← 和块1重叠 DE（防止切断语义）
块3：GHIJK
```

**为什么重叠**：如果一句话正好在切割点，会被切成两半，两边都不完整。重叠让相邻块共享一部分，保证语义连续。

### chunk_size 怎么选

| 大小 | 效果 |
|------|------|
| 太小（100 字） | 语义不完整，检索到的块没上下文 |
| 太大（2000 字） | 一个块含多个主题，检索不精准 |
| **400-600 字** | 平衡点（我们用 400） |

## 三、Embedding：语义检索的基础

### 什么是 Embedding

> 把文本变成一个数字数组（向量），让"语义相近的文本"向量也相近。

```python
"LangChain 是 AI 框架"   → [0.12, -0.34, 0.56, ...]  （512 维）
"LangChain 用来做 AI 应用" → [0.11, -0.33, 0.55, ...]  ← 和上面很接近
"今天天气真好"            → [-0.5, 0.2, -0.1, ...]   ← 完全不同
```

### 相似度计算

向量变成数字后，就能算距离。归一化后**点积 = 余弦相似度**：

```python
def similarity(a, b):
    return sum(x * y for x, y in zip(a, b))  # 点积

similarity(vec_框架, vec_应用)  # 0.886（相关）
similarity(vec_框架, vec_天气)  # 0.292（不相关）
```

**这就是语义检索的数学基础**——不靠关键词匹配，靠"意思相近"。

### 为什么用本地模型

智谱 embedding-2 **收费**（免费档无额度，报 429）。用 `sentence-transformers` 的本地模型：
- 完全免费
- 离线运行（不耗 API 额度）
- 首次下载模型（~100MB），之后离线可用

模型选 `BAAI/bge-small-zh-v1.5`：中文优化、体积小、512 维。

## 四、Chroma：向量数据库

### 为什么需要专门的"向量库"

传统数据库（MySQL/SQLite）存的是结构化数据，查询靠精确匹配（WHERE name = 'x'）。

向量库存的是**向量**，查询靠**相似度**（找最近的几个）：

```python
# 传统数据库
SELECT * FROM docs WHERE title = "报销"     ← 精确匹配

# 向量库
store.search("怎么报销", top_k=3)            ← 语义匹配（"怎么报销"≈"报销流程"）
```

### Chroma 的核心操作

```python
# 存：文本 + 向量 + 元数据
collection.add(
    ids=["doc1_0", "doc1_1"],
    documents=["报销流程...", "考勤制度..."],
    embeddings=[[0.1, ...], [0.2, ...]],
    metadatas=[{"source": "handbook.txt", "doc_id": "doc1"}],
)

# 查：给问题向量，找最相似的 N 个
collection.query(
    query_embeddings=[[0.15, ...]],
    n_results=3,
)
```

### 元数据的作用（重要）

每个向量带 `metadata`（如 `doc_id`、`source`）。这是**增量同步/删除**的基础：

```python
# 按文档删除（更新文档时用）
collection.delete(where={"doc_id": "doc1"})  # 删掉 doc1 的所有向量
```

## 五、RAG 的 Prompt 设计（防幻觉）

```python
RAG_PROMPT = """请根据以下参考资料回答用户的问题。

【要求】
1. 只基于参考资料回答，不要编造资料里没有的内容
2. 如果资料不足以回答问题，诚实说"根据现有资料无法回答"
3. 回答要简洁准确，引用资料里的关键信息

【参考资料】
{context}

【用户问题】
{question}"""
```

**关键指令**：
- "只基于参考资料" → 防止 LLM 用自己的旧知识编造
- "不知道就说不知道" → 防止硬编
- 低 temperature（0.3）→ 让答案稳定、忠实于资料

**这就是 RAG 防幻觉的核心**——不是靠技术，靠 prompt 约束 + 检索提供事实。

## 六、索引同步：RAG 落地的真正难点

### 6.1 问题：文档会变

现在的索引是**手动的**（`python -m rag index`）。真实场景文档会变：

```
运营改了文档 → 没人手动索引 → Agent 还用旧的 → 回答错误
```

### 6.2 四种同步方案

#### 方案 1：定时全量重建（最简单）

```python
# 每天凌晨清空 + 重新索引全部
schedule.every().day.at("02:00").do(reindex_all)
```

- 优点：简单、一定能同步
- 缺点：浪费（99% 没变也重算）
- 适合：文档量小（<1000）、更新不频繁

#### 方案 2：文件监听 + 增量更新

```python
# watchdog 监听文件系统变化
from watchdog import Observer

class Handler(FilesystemEventHandler):
    def on_modified(self, event):
        reindex_single_file(event.src_path)  # 只更新这一个
    def on_deleted(self, event):
        delete_from_vectorstore(event.src_path)  # 删除也处理
```

- 优点：实时、高效
- 缺点：只能监听本地文件
- 适合：本地文件系统

#### 方案 3：数据库 + 时间戳（生产标配）

```python
# documents 表有 indexed_at 字段
# 定时扫描"更新了但没索引的"
changed = db.query("SELECT * FROM documents WHERE updated_at > indexed_at")
for doc in changed:
    vectorstore.delete(filter={"doc_id": doc.id})  # 先删旧
    vectorstore.add(chunk(doc.content), metadata={"doc_id": doc.id})  # 再加新
    db.update(indexed_at=now())
```

- 优点：增量、可溯源、支持删除
- 缺点：需要数据库
- 适合：**生产环境标配**

#### 方案 4：消息队列（大规模）

```
文档系统发现变化 → 发消息到队列 → 索引服务消费 → 更新向量库
```

- 优点：实时、解耦、缓冲、多消费者
- 缺点：架构重
- 适合：大规模（万级文档、高频更新）

## 七、消息队列的真相（重点澄清）

### 7.1 消息队列 ≠ 知道文档变化

> **消息队列只是"快递员/邮局"，它不知道文档变了。** 发现变化是**文档系统**的责任。

```
错误理解：消息队列监控文档 → 发现变化 → 推送
实际：    文档系统发现变化 → 写消息 → 队列搬运 → 索引服务处理
```

### 7.2 谁负责发现变化

| 场景 | 谁发现 | 怎么发现 |
|------|--------|---------|
| 自建系统 | 改文档的代码 | 修改时**主动发消息**（顺手的事） |
| 第三方系统（Confluence） | 第三方 | **Webhook**（改了主动通知你）或轮询 |
| 文件系统 | 你的监听服务 | **watchdog** 监听文件变化 |

**关键**：发现变化的永远是"了解文档状态的组件"，不是消息队列。

### 7.3 消息长什么样

消息就是一条 JSON，发件人和收件人约定格式：

```json
{"doc_id": "doc_001", "action": "updated", "content": "..."}
{"doc_id": "doc_001", "action": "deleted"}
{"doc_id": "doc_002", "action": "created", "content": "..."}
```

### 7.4 删除怎么处理

每个向量带 `doc_id` 元数据。删除 = 按元数据删向量：

```python
elif event["action"] == "deleted":
    vectorstore.delete(filter={"doc_id": event["doc_id"]})
```

### 7.5 为什么要用消息队列（它的真正价值）

不是为了"发现变化"，而是：
- **可靠**：下游挂了消息不丢（队列存着，恢复后继续）
- **缓冲**：突然 1000 条消息，排队慢慢处理不崩
- **解耦**：发件人不用管收件人状态
- **多消费者**：一条消息被索引服务、搜索服务、通知服务分别消费

**系统简单时（单机、低并发），直接调函数就行，不需要队列。**

## 八、完整生产架构

```
┌────────────────┐
│ 文档系统        │ ← 发现变化（应用/Webhook/监听）
└───────┬────────┘
        │ 发消息（doc_id, action, content）
        ▼
┌────────────────┐
│ 消息队列        │ ← 只暂存和投递（Kafka/RabbitMQ）
└───────┬────────┘
        │
        ▼
┌────────────────┐
│ 索引服务        │ ← 消费消息，更新向量库
│ created → 加   │
│ updated → 删+加│
│ deleted → 删   │
└───────┬────────┘
        │
        ▼
┌────────────────┐
│ 向量库          │ ← 每个向量带 doc_id
└────────────────┘
```

## 九、踩坑记录

### 🕳️ 踩坑 1：智谱 embedding-2 余额不足

**现象**：调 `client.embeddings.create` 报 429 "余额不足"。

**原因**：智谱免费档 **不含 embedding 额度**（和 web_search 一样是收费工具）。

**解决**：换本地模型 `sentence-transformers`（BAAI/bge-small-zh-v1.5），免费离线。

**教训**：**免费档的限制要提前查清**。"模型免费" ≠ "所有功能免费"。Day 3 踩过 web_search 的坑，这次是 embedding。

### 🕳️ 踩坑 2：包内 import 找不到模块

**现象**：`rag/store.py` 里 `from rag.loader import Chunk` 报 `ModuleNotFoundError`。

**原因**：直接跑 `python rag/store.py` 时，Python 不知道 `rag` 是包。

**解决**：文件顶部加 `sys.path.insert(0, 项目根)`，或用 `python -m rag.xxx`。

**教训**：Day 5 踩过同样的坑。Python 包内 import 有固定规矩。

## 十、难点与思考

### 思考 1：RAG 的工程难点不在"检索"，在"同步"

> 很多人以为 RAG 难在 Embedding 算法或向量库调优。
> **真正落地时，80% 工程量在"索引同步"**：文档怎么入库、变了怎么发现、删除怎么清理、大量文档怎么高效处理。
>
> 这就是为什么 Dify/Coze 这些平台的核心能力不是"RAG 算法"，而是"文档管理"。

### 思考 2：RAG 的幻觉防线在 prompt

> RAG 不能 100% 消除幻觉，但能大幅降低。
> 防线是 prompt 约束（"只基于资料""不知道就说不知道"）+ 检索提供事实。
> **LLM 还是可能编**，所以关键场景仍需人工复核。

### 思考 3：本地 Embedding 的取舍

| 维度 | 本地模型 | API（智谱/OpenAI） |
|------|---------|------------------|
| 成本 | 免费 | 按 token 付费 |
| 速度 | 快（本地 GPU/CPU） | 受网络影响 |
| 质量 | 中等（小模型） | 好（大模型） |
| 部署 | 要下载模型 | 零部署 |

学习用本地，生产看场景（质量优先用 API，成本优先用本地）。

### 思考 4：Agent 自主工具选择——从"能用工具"到"会选工具"

把 RAG 接入 Agent 后，做了一个关键测试：**Agent 会不会自主选择 query_docs？**

测试结果非常有说服力：

```
【私有问题】"我们公司的报销流程是什么"
  Agent 行为：
    1. 先调 search_web（搜互联网）
    2. 发现互联网搜不到公司内部流程
    3. 自主切换到 query_docs（查本地知识库）
    4. 准确回答 5 步流程 + 5000 元 VP 审批

【公开问题】"LangChain 是什么"
  Agent 行为：
    1. 调 search_web + fetch_url
    2. 没用 query_docs（公开信息不需要查本地，决策正确）
```

**这个测试印证了 Day 4 笔记"思考 5"的核心观点**：

> **工具之间的切换决策，是 Agent 智能的体现。** 工具数量不是关键，工具之间能否形成"决策链"才是。

Agent 不是机械地"给什么用什么"，而是会"看结果，换策略"——互联网搜不到就转查知识库。这是从"能用工具"到"会选工具"的质变。

**关键设计**：让 Agent 会选工具的秘诀在 schema 的 `description`：
- search_web：强调"互联网公开信息"
- query_docs：强调"私有文档，互联网搜不到的要靠这个"
- 两者的 description 形成"互补关系"，LLM 读后自然知道什么时候用哪个

> 🔑 **好的工具描述 = 给 LLM 的"选择指南"。** 不是列功能，是教它"什么时候该用我"。

### 思考 5：消息队列的真相——它是"快递员"，不是"监控者"

讨论索引同步时，对"消息队列方案"产生了关键澄清：

**误区**：看到"文档变了 → 发消息 → 队列 → 更新向量库"的架构图，容易以为**消息队列在监控文档变化**。

**真相**：**消息队列只是"快递员/邮局"，它不发现变化，只负责搬运消息。** 发现变化是**文档系统**的责任。

| 谁发现变化 | 怎么发现 | 场景 |
|-----------|---------|------|
| 改文档的代码 | 修改时**主动发消息**（顺手的事） | 自建系统 |
| 第三方系统 | **Webhook**（改了通知你）或轮询 | Confluence/Notion |
| 你的监听服务 | **watchdog** 监听文件系统 | 纯文件 |

**消息队列的真正价值**（不是"发现变化"，而是）：
- **可靠**：下游挂了消息不丢（队列存着，恢复后继续）
- **缓冲**：突然 1000 条消息，排队慢慢处理不崩
- **解耦**：发件人不用管收件人状态
- **多消费者**：一条消息被多个服务分别消费

**一句话**：系统简单（单机、低并发）时直接调函数就行，不需要队列。队列解决的是"规模和解耦"问题，不是"发现变化"问题。

> 🔑 这个澄清很重要——很多人被架构图误导，以为引入消息队列就能"自动发现文档变化"。实际上发现变化的机制（Webhook/监听/主动发）才是核心，队列只是传输层。

### 思考 6：为什么"不让用户直接改文件"是生产 RAG 的隐藏前提

讨论同步方案时发现一个反直觉的真相：

```
❌ 让用户直接改文件系统 → 难追踪变化 → 同步复杂
✅ 用户通过系统改文档（API/Web）→ 系统知道每次变化 → 同步简单
```

这就是为什么企业 RAG 都要配"文档管理系统"——不是为了文档管理，是为了**让变化可追踪**。

- 用户直接改文件：你只能靠 watchdog 轮询发现（被动、延迟）
- 用户通过系统改：系统在修改的瞬间就知道（主动、实时）

**推论**：RAG 的同步难度，取决于"文档变更是否经过你掌控的系统"。掌控越强，同步越简单。

## 十一、关键概念速查表

| 术语 | 含义 |
|------|------|
| **RAG** | 检索增强生成，让 LLM 基于私有文档回答 |
| **Chunk** | 文档切块，每块约 400-600 字 |
| **Embedding** | 文本转向量，让语义相近的文本向量相近 |
| **余弦相似度** | 衡量两个向量方向的相似度（归一化后=点积） |
| **向量库** | 专门存/检索向量的数据库（Chroma/Milvus） |
| **top_k** | 检索时取最相似的 k 个块 |
| **indexed_at** | 标记文档是否已同步到向量库（增量更新用） |
| **doc_id 元数据** | 每个向量带 doc_id，便于按文档删除/更新 |
| **消息队列** | 只负责暂存和投递消息（**不发现变化**，只是快递员） |
| **Webhook** | 第三方系统在变化时主动通知你（发件人主动） |
| **watchdog** | 监听文件系统变化的库（被动发现） |
| **query_docs** | Agent 查本地知识库的工具（RAG 接入 Agent） |
| **工具切换决策** | Agent 看结果换策略（互联网不行→查知识库） |

```
data/chroma/        向量库索引（运行时生成，gitignore）
```

**RAG 有两种用法**：
1. **独立 CLI**：`python -m rag ask "问题"`（纯 RAG 问答，不经过 Agent）
2. **Agent 工具**：Agent 自主决定调用 `query_docs`（和其他工具协作）

## 十三、亲手体验

```bash
# 1. 索引文档
python -m rag index

# 2. 单次问答
python -m rag ask "报销流程是什么？"

# 3. 交互式问答
python -m rag
# 输入：Git 提交信息怎么写？
# 输入：quit 退出
```
